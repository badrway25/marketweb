# Corporate-suite distinctness matrix

**Status**: operational v1 · **Date**: 2026-04-28
**Purpose**: turn distinctness from a vague idea into a row-by-row, dimension-by-dimension audit. Used at intake (planner) and at critique (style-critic) to triangulate the next sibling against Pragma + Fiscus + Solaria.
**Companion files**: `corporate-suite-reference-pack.md` · `next-template-brief-schema.md`.
**Source standards**: `DISTINCTNESS_RULES.md` (4-of-5-axes gate) · `factory/standards/corporate-suite-design-standard.md` (rule IDs).

---

## 0 · How to read this matrix

For each dimension below:
- **Pragma / Fiscus / Solaria column** = what is locked in today. The next sibling cannot collide on this cell.
- **What is already distinct** = where the three siblings successfully differ. Inherit the differentiation strategy.
- **Where overlap is too high** = the load-bearing risk. Where the next sibling is most likely to silently collapse into a sibling rename.
- **What the next sibling MUST NOT repeat** = explicit ban list. The planner brief copies this row verbatim into the new template's docstring as anti-drift binding.

A row that says "MUST NOT repeat: [Pragma's KPI tuple]" means: the new sibling's planner cannot ship a hero meta-strip composed of years/clients/€/% — it must invent a new proof-shape.

The matrix is checked at four gates: plan (A.2) · pack (A.3) · critique (A.6) · walk (A.7). Each gate reads this matrix as the source of truth for "is this still distinct enough."

### 1.1 · Tone

| | Pragma | Fiscus | Solaria |

| Tone label | advisory-sober (boardroom) | institutional-fiscal (presidio) | professional-warm (method-declared) |

| Stance | strategy-first · decisional gravity | scadenze-first · adempimento corretto | bounded percorso · method-declared |

**Already distinct**: three different stances on the firm-client relationship (gatekept · presidio · bounded). Three different registers within the same `Lei/noi` polite Italian.

**Overlap risk**: all three are "serious Italian professional voices." A new sibling defaulting to the same tonal band reads as a fourth flavor of the same drink. Differentiation must come from a *new* stance, not a *new* synonym for the same stance.

**Next sibling MUST NOT repeat**:
- Pragma's "decisional gravity" framing
- Fiscus's "presidio + scadenze-first" framing
- Solaria's "non-terapia non-consulenza" bounded-method framing
- Any voice anchor that depends on a contrast pair if Solaria's already does

**Open stances**: stewardship (notary/fiduciary) · evidence-first (legal litigation) · longitudinal-care (revisione legale, multi-year audit) · custodial (asset management) · public-trust (independent director) · architectural-discipline (design firm).

## 2 · How to use the matrix at intake

At workflow A.1 (intake), the orchestrator fills out a **draft row** for the new sibling across all 12 dimensions, alongside the three existing rows. Then:

1. For each dimension, mark **distinct (✓)** or **collides (✗)** vs each existing sibling.
2. Tally axes-different per existing sibling (e.g., "vs Pragma: 4/5 · vs Fiscus: 5/5 · vs Solaria: 5/5").
3. Apply DISTINCTNESS_RULES §1: **≥ 4 of 5 axes must differ vs every existing sibling** (the five axes there are voice · palette · imagery · typography · structure — they map to dimensions 1, 2-3, 6, 4, 8 here).
4. If any pair scores ≤ 3/5, re-spec the offending dimensions before plan sign-off.
5. The filled matrix is appended to the planner brief and copied verbatim into the new template's content-module docstring (CS-EXEC-02).

## 4 · Matrix maintenance

- Each new sibling adds a column to this matrix at draft-landing (workflow A.9).
- Each closed-out edit pass (workflow B) that changes any of the 12 dimensions updates the relevant cell.
- The matrix is the orchestrator's source of truth for "what is taken" and supersedes any individual template's docstring if they disagree.

When a 4th sibling lands, this matrix grows from 3 to 4 columns. The "what the next sibling MUST NOT repeat" lines are extended, NOT replaced — the cluster's prohibition list is monotonic.
