Django renders 500 outside the normal template-context pipeline: the request-context processors are NOT run (the request is in a failed state, so resolving them could trigger another exception loop). That means no `request`, no `user`, no `csrf_token`, no `messages`, and — critically — no resolution of `{% static %}` via the static-files-finder when the project is in DEBUG=False and STATIC_ROOT has not been collected yet. Two consequences for the design: 1. We render a self-contained HTML document with inlined CSS, no Django URL reverse, no template inheritance. 2. Navigation links are hard-coded to "/" and "/templates/" — these are the canonical roots of the marketplace and never moved across sprints. {% url %} would risk a second exception if a URLConf import is itself the source of the 500. Auto-picked by Django when DEBUG=False; dev still shows the yellow debug 500 (with traceback) so the dev workflow stays intact.
